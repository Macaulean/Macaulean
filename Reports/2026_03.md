## Wins

### CAS architecture redesign: complete and extensible

The entire CAS backend was redesigned and implemented around a three-layer abstraction: mathematical objects, computational problems, and CAS results, communicated via the MRDI (Mathematical Result Data Interchange) protocol. The architecture features strategy-based dispatch (modeled on Lean's `grind` solver extensions), lazy process lifecycle management, hash-based caching, and per-request timeouts.

This design means that adding a new CAS operation requires only (1) defining new MRDI types, (2) implementing a backend handler, and (3) registering a strategy. No core code changes are needed.

### Three working CAS backends

The project went from one backend (Macaulay2) to three:

* **Macaulay2** -- polynomial reduction, integer factorization.
* **SymPy** (Python) -- adds Groebner basis computation, polynomial factorization, radical membership.
* **Oscar/Julia** -- adds permutation group membership via GAP.

The SymPy backend was added specifically to validate the abstraction: a second backend plugged in without modifying core code. Oscar extended this further to a completely different domain (finite group theory).

### Five verified mathematical strategies, all kernel-checked with zero axioms

1. **Ideal membership** -- CAS provides quotients, Lean verifies `f = sum q_i * g_i`.
2. **Groebner basis reduction** -- compute a Groebner basis first, then reduce.
3. **Radical membership** -- Rabinowitsch trick; CAS certifies a power and quotients.
4. **Polynomial divisibility** -- CAS factors, kernel verifies cofactor.
5. **Permutation group membership** -- GAP finds a generator word, kernel verifies composition via `decide`.

In every case the CAS is used purely as a data source; the Lean kernel owns all correctness.

### Custom polynomial data structures for efficient verification

A significant strand of work this quarter went into designing polynomial representations that more closely mirror Macaulay2's internal format, enabling more efficient equality checking and, critically, more efficient kernel verification of certificates. This included a new sorted term-list representation with verified monomial reordering and merging, as well as handling of integer coefficients. These data structures underpin both the `algebra_norm` tactic and the ideal membership pipeline.

### AlgPoly: sorry-free reflective algebra normalization

A two-level polynomial type (`AlgPoly`) with parameterized coefficients was built from scratch, fully verified (all `sorry` eliminated), and integrated as the `algebra_norm` tactic. All 10 test cases produce kernel-checked proofs.

### Permutation group verification with serious test cases

Membership in A_5 (order 60, smallest non-abelian simple group), S_5 (order 120), and D_4 (dihedral group of the square) is verified by the kernel with zero axioms. This is, to our knowledge, the first kernel-verified permutation group membership tactic in Lean backed by a CAS.

AI coding tools (Claude and Codex) were used extensively this quarter, co-authoring 27 of 108 commits (25%). This was not limited to exploratory experiments: AI assistants implemented the CAS architecture redesign (strategy dispatch, MRDI protocol, backend lifecycle), both new backends (SymPy, Oscar), several new tactics (m2polyrith, gb_reduce, radical membership, polynomial divisibility), and the full AlgPoly sorry elimination. Earlier in the quarter, AI-generated Macaulay2 code produced the initial `m2sos` (sum-of-squares) tactic, which -- while requiring correction -- prompted useful discussions about expression restructuring and its effect on Groebner basis behaviour and certificate size.

### Efficient Lean term generation

The team explored multiple approaches to more efficient Lean term generation, with particular attention to managing memory usage and kernel recursion depth -- a nontrivial engineering challenge given Lean's kernel architecture. Test cases now include computationally hard problems: 16-term polynomials in 4 variables, cyclotomic polynomial divisibility, and large polynomial ideals.

## Challenges

### `native_decide` axiom violation (resolved)

The initial permutation group membership implementation used `native_decide`, which introduces `Lean.ofReduceBool` -- an axiom that violates the project's core principle that the kernel alone owns correctness. This was resolved by redesigning the strategy: the CAS now returns concrete permutation images rather than abstract words, and `decide` (kernel-only) verifies composition.

### Kernel recursion depth and memory limits

Large polynomial certificates stress Lean's kernel evaluator. One benchmark (a 9-variable polynomial identity, cf. leanprover/lean4#11861) remains deferred as it exceeds current kernel capacity. This is a known upstream limitation, not a project bug, but it constrains the size of problems we can certify today.

### Certificate size

Even when the kernel can handle a certificate, large Groebner bases and long quotient expressions slow verification. A recurring discussion theme was finding methods for obtaining smaller Groebner bases and shorter certificates, and understanding how expression restructuring and choice of monomial order affect tractability. Reducibility boundaries -- what to unfold and what to keep opaque -- also required careful tuning.

### CAS architecture iteration

The original CAS design (closed `TaskKind` enum, ad-hoc RPC wiring) did not scale to multiple backends or new operations. This required a full redesign mid-quarter (strategy dispatch, open MRDI type vocabulary, framework-managed process lifecycle). The redesign is complete and the resulting architecture is significantly more extensible, but the iteration cost was nontrivial.

### Grind integration exploration (deferred)

Early-quarter work explored integrating algebra normalization into Lean's `grind` framework via solver extensions and annotations. This was abandoned after discovering that `pushEq` interferes with Groebner basis reasoning and that annotation-based approaches were insufficient for our needs. The standalone `algebra_norm` tactic was built instead and is working well.

## Needs

### Lean kernel performance improvements

The biggest bottleneck is kernel evaluation of large polynomial certificates. Upstream improvements to recursion depth limits and memory handling would directly unlock harder problem instances. Engagement with the Lean core team on this would be valuable.

### CI/CD infrastructure

The project currently has no automated testing pipeline. As the codebase grows across three CAS backends, a CI system that can install Macaulay2, SymPy, and Oscar, then run `lake build MacauleanTest`, would catch regressions early and increase contributor confidence.

### User-facing documentation

The architecture documents (CAS_ARCHITECTURE.md, CAS_REDESIGN.md) are thorough but developer-facing. User-facing documentation for how to install backends and use the tactics beyond Macaulay2 is minimal.

### Support for non-Q coefficient fields

The current pipeline is limited to characteristic-0 computations (integers and rationals). Expanding to finite fields (ZMod p) would both unlock new mathematical applications and offer potential speed-ups by working modulo primes. This is planned but not yet implemented.
